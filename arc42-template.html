1. Einführung und Ziele

1.1. Aufgabenstellung

Bei dem Projekt handelt es sich um ein Lernprojekt. Mit Hilfe einer öffentlichen REST Schnitstelle werden Daten von einem fremden System abgerufen. Diese Daten werden nach dem Abrufen mit einer eigens entwickelten Geschäftslogik transformiert, angereichert und für Prüfungen verwendet. In der Abschlusspräsentation wird die Geschäftslogik anhand eines UseCase präsentiert.

1.2. Qualitätsziele

Ziel Erläuterung

Lauffähige Software

Das Projekt wird in einer Live-Demo auf einem Rechner gezeigt. Client des Projektes ist entweder der Browser oder z.B. Discord.

github-Projekt

3. Kontextabgrenzung

Dieses Kapitel beschreibt die Abgrenzung des Systems zu anderen und die Kommunikationsbeziehungen.

3.1. Fachlicher Kontext

Benutzer

Das System kann über eine grafische Oberfläche aufgerufen werden. Der Benutzer kann daraufhin ein Zitat abrufen und dieses bewerten. Das System übernimmt dabei die Wahl eines Zitates und empfielt eine Bewertung.

Tronald Dump API

Mithilfe der Tronald Dump API werden vom System Zitate von Donald Trump abgerufen. Das System erwartet entsprechend der mitgegebenen Paramter ein zutreffendes Zitat.

Bewertung

Es existiert bereits Wissen zu der Bewertung von Zitaten. Zusammen mit den neuen erkenntnissen legt das System Daten an. Diese können zukünftig auch für andere Systeme genutzt werden.

4. Lösungsstrategie

Thechnologiestrategien

Am Anfang war es wichtig, sich Gedanken zu machen, mit welchen Technologie man das Projekt umsetzten will, da diese das Projekt Grundlegend beeinflussen. Hier wurde sich für Spring Boots entschieden, da die Vorteile gegenüber den Nachteilen stark dominierten. Des Weiten verbargen sich für uns auch kleine, zeitliche Vorteile in den Nachteilen.

Vorteil Nachteil

Hohe Flexibilität

Arbeit mit Java

Einfache Implementierung

Kaum Einarbeitung

Gute Dokumentation

Ein Nachteil von Spring Boot ist, dass es mit Java arbeitet und es daher zu schlechten Laufzeiten kommt. Jedoch wurde sich bewusst für Spring Boot entschieden, da Java eine Programmiersprache ist, die allen Projektteilnehmern bekannt war. Dies war im Hinblick auf die vorgegeben Zeit, zur Umsetzung des Projektes, sehr essenziell, da sich so die Einarbeitungszeit in eine neue Programmiersprache erspart wurde.

Lösungsstrategie

Ziel unseres Systems sollte es sein, dass der Benutzer durch Zufall einen Spruch zurückbekommt, wenn er einen Knopf drückt. Der Zufall sollte mit Hilfe eines Algorithmus generiert werden. Dieser sollte nach einem Schema arbeiten, das die Merkmale Alter und die Anzahl der Wörter berücksichtigt. Je älter der Spruch ist, desto höher ist seine Gewichtung. Gleiches mit den Anzahl der Wörtern. Je mehr Wörter ein Spruch besitzt, desto höher wird er angesiedelt. Übergibt der Benutzer dem System den Befehl, dass dieser gerne einen neuen Spruch hätte, schickt das System der Api eine Anfrage. Die Api sucht, mit Hilfe des Algorithmus, einen Spruch heraus und schickt diesen weiter an das System. Das System nimmt den Spruch an und übergibt ihn am Ende dem Nutzer.

Organisatorische Entscheidungen

Um das Projekt präzise zu gestalten, haben wir uns dafür entschieden, dieses zu unterteilen. Hierbei wurde das Projekt in drei Teilbereiche: „Entwicklung API“, „Entwicklung Backend“ und der Dokumentation „Arc42“ aufgeteilt. Uns war es wichtig, dass jeder in der Gruppe Wissen über alle Aktivitäten erlangt. Deshalb haben wir uns in regelmäßigen Abständen getroffen und uns über unsere Arbeit ausgetauscht.

6. Laufzeitsicht

Diese Sicht erklärt den Ablauf einer Benutzerabfrage.

  • Der Benutzer startet den Service durch Aufrufen der Domain in einem Browser.

  • Das System ruft über die Schnittstelle von der Tronald Dump API zufällige Zitate von Donald Trump ab. Dafür wird die Schnittstelle mehrfach angesprochen.

  • Anhand des Zitates und der Jahreszahl berechnet der Algorithmus des System dn Schadenersatz. Dafür wird zunächst die Anzahl an Wörtern mit dem alter in Jahren multiplizirt. anschließend wird noch eine statische Gewichtung des Systems multipliziert. Diese Gewichtung stammt aus dem Wissen des Systems und der Entwickler.

  • Die vom Sytsem erstellte Liste mit den Zitaten und dem Schadenersatz wird anschließend dem Benutzer im Browser angezeigt. Bei Störungen wird der entsprechende Error zurückgeliefert.
